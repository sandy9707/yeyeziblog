服务器加装硬盘备份迁移

Created2025-04-04|Updated2025-04-04|TechLinux
|Post View:

引言

这篇文章可以帮助系统管理员或需要进行服务器数据迁移的用户。下面是一个使用 rsync 进行服务器数据备份的 文章大纲和核心内容：

使用 rsync 将旧服务器数据安全高效地备份到新服务器的新硬盘。

效果展示

过程

  1. 引言

    • 数据备份的重要性（尤其在服务器更换、硬件升级时）
    • 场景描述：有一台旧服务器（Source Server）和一台新服务器（Destination Server），新服务器已安装新硬盘并准备就绪。
    • 为什么选择 rsync？（高效、增量备份、保持文件属性、灵活等优点）

  2. 准备工作 (Prerequisites)

    • 网络连接： 确保两台服务器之间网络互通（可以通过 ping 测试）。
    • SSH 访问：
      • 确保新服务器可以 SSH 登录到旧服务器。
      • 强烈推荐： 配置 SSH 密钥对认证（免密码登录），这对于自动化和长时间运行的 rsync 任务至关重要且更安全。
        • 在新服务器上生成密钥对：ssh-keygen -t rsa
        • 将公钥复制到旧服务器：ssh-copy-id user@source_server_ip
    • 用户和权限：
      • 确定在旧服务器上拥有读取源数据所需权限的用户。
      • 确定在新服务器上拥有写入目标路径所需权限的用户。
    • 目标路径：
      • 在新服务器上，确保新硬盘已正确分区、格式化并挂载到指定的目录（例如 /backup 或 /data）。使用 df -h 确认挂载点和可用空间。
      • 创建用于存放备份数据的目标文件夹，例如 mkdir -p /backup/source_server_data。
    • 明确备份内容： 确定旧服务器上需要备份的具体目录路径（例如 /var/www/html, /home, /etc 等）。

  3. 核心工具：rsync 详解

    • rsync 基本语法：rsync [OPTIONS] SOURCE DESTINATION
    • 常用且关键的选项 (OPTIONS) 解析：
      • -a (archive): 归档模式，相当于 -rlptgoD 的组合。
        • -r: 递归复制目录。
        • -l: 复制符号链接本身。
        • -p: 保留文件权限。
        • -t: 保留文件修改时间。
        • -g: 保留文件所属组。
        • -o: 保留文件所有者（仅 root 用户有效）。
        • -D: 保留设备文件和特殊文件（仅 root 用户有效）。
      • -v (verbose): 显示详细输出，让你看到哪些文件正在被传输。
      • -h (human-readable): 以 K, M, G 等易读格式显示文件大小等信息。
      • -P: 相当于 --progress --partial。
        • --progress: 显示每个文件的传输进度。
        • --partial: 保留传输中断的部分文件，下次运行时可以续传，非常适合大文件和不稳定网络。
      • --delete: （谨慎使用！） 删除目标服务器上存在但源服务器上不存在的文件。这能让目标目录成为源目录的精确镜像，但如果源或目标路径指定错误，可能会误删数据。首次备份可以不加，后续同步时考虑使用。
      • --exclude=PATTERN: 排除不需要备份的文件或目录（例如缓存文件、临时文件）。可以多次使用。示例：--exclude='cache/' --exclude='*.tmp'
      • --bwlimit=KBPS: 限制传输带宽（单位 KB/s），避免占用过多网络资源影响其他服务。
      • -e ssh: 指定使用 SSH 协议进行传输（通常是默认的，但明确指定更清晰）。
      • -n (dry-run): 测试运行，显示会执行哪些操作但不实际传输或删除任何文件。强烈建议在正式执行前（特别是使用 --delete 时）先用 -n 选项检查一遍。

  4. 执行备份操作 (Step-by-Step)

    • 登录目标服务器（新服务器）： 这是推荐的操作方式，从目标服务器“拉取”数据。

    • 构建 rsync 命令：

      • 基本备份命令（首次备份，不删除目标多余文件）：

      • 带排除和删除的同步命令（后续更新备份）：

    • 重要提示：源路径末尾的斜杠 /

      • source/：表示复制 source 目录下的 内容 到目标目录。
      • source：表示复制 source 目录 本身 到目标目录。
      • 根据你的需求选择是否添加末尾斜杠。 大多数备份场景是复制目录内容，所以源路径末尾通常带 /。

    • 执行命令： 粘贴并运行你构建好的 rsync 命令。首次备份可能需要较长时间，取决于数据量和网络速度。

    • 使用 screen 或 tmux： 对于长时间运行的备份任务，强烈建议在 screen 或 tmux 会话中运行 rsync 命令，这样即使 SSH 连接意外断开，备份任务也能在后台继续进行。

      • 启动 screen: screen -S backup_session
      • 运行 rsync 命令…
      • 分离会话：按 Ctrl+A 然后按 D
      • 重新连接会话：screen -r backup_session

  5. 验证备份结果 (Verification)

    • 检查 rsync 输出： 命令执行完毕后，rsync 会报告传输的总数据量、速度等信息。检查是否有错误信息。

    • 对比文件数量和大小：

      • 在源服务器上：ls -lR /path/to/source | wc -l 和 du -sh /path/to/source
      • 在目标服务器上：ls -lR /path/to/destination | wc -l 和 du -sh /path/to/destination
      • 比较两者结果是否大致相符（注意 du 的结果可能因块大小略有差异，文件数是更可靠的指标之一）。

    • 抽查文件： 随机选择一些重要或代表性的文件/目录，检查它们是否存在于目标服务器，并且内容、权限、时间戳是否正确。

    • 使用 rsync 的 -n (dry-run) 再次检查： 备份完成后，立即再运行一次带有 --delete（如果计划使用）和 -n 的 rsync 命令。理想情况下，它应该报告没有文件需要传输或删除（或只有少量预期中的变动）。

  6. 注意事项与排错 (Notes and Troubleshooting)

    • 权限问题： 确保执行 rsync 的用户在源端有读取权限，在目标端有写入权限。使用 -a 选项会尝试保留权限和所有权，如果目标端用户不是 root，可能无法完全保留所有者/组信息。
    • 磁盘空间： 确保目标服务器的新硬盘有足够的空间容纳所有备份数据。
    • 防火墙： 检查两台服务器上的防火墙（如 iptables, firewalld, ufw）是否允许 SSH（默认端口 22）连接。
    • SELinux/AppArmor： 如果启用了这些安全模块，可能需要配置规则允许 rsync 或 SSH 的相关操作。
    • --delete 风险： 再次强调，使用 --delete 前务必确认源和目标路径无误，最好先用 -n 测试。
    • 大量小文件： rsync 在处理海量小文件时，性能瓶颈可能在于文件元数据的检查和设置，而不是网络带宽。耐心等待或考虑优化（如先打包再传输，但会失去增量备份的优势）。

    • 确保使用了 SSH 密钥对免密登录。

    • 重申 rsync 是一个强大而灵活的数据备份和同步工具。
    • 强调准备工作、仔细构建命令、使用 -n 测试以及备份后验证的重要性。
    • 鼓励根据实际需求调整 rsync 选项，并考虑自动化备份方案。

1. 使用 rsync 备份所有用户数据

通常，Linux 系统上的用户家目录位于 /home 目录下。备份 /home 目录是备份用户数据的主要部分。

命令选项解释:

  • sudo: rsync 需要 root 权限才能读取所有用户的文件并保留原始的所有权和权限信息。
  • rsync: 备份工具本身。
  • -a (archive): 归档模式，相当于 -rlptgoD 的组合：
    • -r: 递归处理子目录。
    • -l: 保留符号链接。
    • -p: 保留文件权限。
    • -t: 保留文件修改时间。
    • -g: 保留文件所属组 (group)。
    • -o: 保留文件所有者 (owner)。这是保留用户归属的关键！
    • -D: 保留设备文件和特殊文件。
  • -A: 保留 ACL (访问控制列表)，如果你的系统使用了 ACL。
  • -X: 保留扩展属性 (Extended Attributes)。
  • -v (verbose): 显示详细输出，让你看到哪些文件正在被传输。
  • --delete: 删除目标目录中存在但源目录中已不存在的文件。这使备份目录成为源目录的精确镜像。使用此选项要小心，确保你理解它的作用！ 如果你只想增量添加文件而不删除目标目录多余的文件，请不要使用此选项。
  • -e ssh (仅远程备份): 指定使用 SSH 协议进行传输。这是远程备份的标准做法。
  • /home/: 源目录。注意末尾的 / 很重要。它表示复制 /home 目录的 内容 到目标目录。如果没有末尾的 / (/home)，它会把 home 目录本身复制到目标目录下，形成 /path/to/backup/home/home/。
  • /path/to/local/backup/home/ 或 user@remote_server:/path/to/remote/backup/home/: 目标目录。

其他常用选项:

  • --exclude=PATTERN: 排除符合模式的文件或目录。例如，排除所有用户的缓存目录：--exclude='*/.cache/'。可以多次使用 --exclude。

  • --progress: 显示每个文件的传输进度。

  • -z: 在传输过程中压缩文件数据，可以节省带宽，但会增加 CPU 负担。

  • -H: 保留硬链接。

  • --dry-run 或 -n: 测试运行，显示会执行哪些操作，但实际上不进行任何更改。在正式运行前用这个选项检查命令是否符合预期非常有用。

执行备份:

  1. 选择目标: 确定备份是存储在本地另一个磁盘 (/path/to/local/backup/home/) 还是远程服务器 (user@remote_server:/path/to/remote/backup/home/)。确保目标位置有足够的空间。
  2. 配置 SSH (远程备份): 如果是远程备份，确保源服务器的 root 用户（或者你用来执行 sudo 的用户）可以通过 SSH 免密登录到目标服务器的 user 账户。这通常通过设置 SSH 密钥对来实现 (ssh-keygen, ssh-copy-id)。
  3. 运行命令: 在源服务器上以 root 权限执行 rsync 命令。

2. 备份后用户归属变了怎么办？

这是 rsync 备份中一个常见且重要的问题。rsync -o -g (包含在 -a 中) 会尝试保留文件的数字 UID (用户 ID) 和 **GID (组 ID)**。

问题根源:

用户归属看起来 “变了” 的主要原因是：源服务器和目标服务器上的用户/组名称与数字 UID/GID 的映射关系不一致。

  • 场景 1: UID/GID 不存在于目标服务器: 如果源服务器上文件属于 UID 1005，但目标服务器上没有 UID 为 1005 的用户，那么在目标服务器上查看文件时，所有者会显示为数字 1005 而不是用户名。
  • 场景 2: UID/GID 在目标服务器上属于不同用户: 这是更麻烦的情况。如果源服务器上 userA 的 UID 是 1001，文件属于 userA。但在目标服务器上，UID 1001 可能属于 userB (或者根本不存在)。rsync 会将文件的所有者设置为 UID 1001。如果你在目标服务器上查看，文件会显示为属于 userB (如果 userB 存在)，或者显示为数字 1001。

解决方案:

按推荐程度排序：

  1. 【最佳实践】保持源和目标服务器用户/组 ID 一致:

    • 这是最根本、最可靠的解决方案。确保两台服务器上的用户名和对应的 UID、组名和对应的 GID 完全相同。
    • 实现方法:
      • 手动同步: 在创建用户和组时，手动指定相同的 UID 和 GID (使用 useradd -u <UID>, groupadd -g <GID>)。对于已存在的用户/组，可以使用 usermod -u <newUID> 和 groupmod -g <newGID> 修改，但这比较危险，需要确保没有进程以旧 UID/GID 运行，并可能需要更新系统中文件的所有权 (find / -uid <oldUID> -exec chown <newUID> {} \; 等)。
      • 集中式用户管理: 使用 LDAP、NIS 或其他目录服务来统一管理所有服务器上的用户和组信息。这是企业环境中的标准做法。
    • 优点: rsync -a 可以完美工作，备份和恢复后的所有权完全正确，无需额外步骤。

  2. 使用 --numeric-ids 选项:

    • rsync --numeric-ids ...
    • 这个选项告诉 rsync 不要尝试根据名称映射 UID/GID，直接使用源端的数字 UID/GID 写入目标端。
    • 注意: rsync 在以 root 身份运行时，-o 和 -g (包含在 -a 中) 通常默认就像使用了 --numeric-ids 一样工作，即它们倾向于保留数字 ID。但明确指定 --numeric-ids 可以确保此行为，尤其是在跨不同系统或 rsync 版本时。
    • 局限性: 这并不能解决根本的 ID 不匹配问题。它只是确保了数字 ID 被正确复制。在目标服务器上查看文件时，如果 UID/GID 映射不同，你仍然会看到 “错误” 的用户名或只有数字 ID。但在需要恢复到 原始服务器 或 另一个 ID 映射与原始服务器相同的服务器 时，所有权将是正确的。

  3. 【较复杂】备份后在目标端修复所有权:

    • 如果无法统一 UID/GID，可以在 rsync 完成后，在目标服务器上运行一个脚本来修复所有权。

总结

  • 使用 sudo rsync -aAXv /home/ destination/ 是备份用户数据的良好起点。根据需要添加 --delete, --exclude, -H 等选项。
  • 用户归属问题的最佳解决方案是确保源服务器和目标服务器具有一致的 UID/GID 映射，最好通过集中式用户管理实现。
  • 如果无法统一 ID，rsync (特别是带 -a 或 --numeric-ids) 会保留数字 ID。这对于恢复到原始服务器或 ID 相同的服务器是有效的，但在目标服务器上查看时所有权可能显示不正确。
  • 备份后修复所有权是一个复杂且不太推荐的后备方案。

在执行任何操作之前，尤其是在使用 --delete 或运行修复脚本时，请务必 进行测试 (例如使用 --dry-run) 并确保你有可靠的数据备份。
